Prep the zip

Both destinations consume the same artifact: a zip whose only top-level entry is your plugin folder. Get this right once and the rest is copy-paste.

Folder layout

The checker rejects zips that unpack loose files at the root. Everything lives inside one directory named after the plugin.

# inside the zip, exactly one top-level folder:
floating_info_tool/
├── __init__.py        # must expose classFactory(iface)
├── metadata.txt       # the manifest — see below
├── plugin.py          # your main module
├── icon.png           # toolbar/manager icon
├── LICENSE            # GPL-compatible for the official repo
└── resources.py       # compiled from resources.qrc, if used

metadata.txt — the manifest

This file is the source of truth for version, compatibility and everything Plugin Manager displays. The repo parses it on upload.

name
Display name in Plugin Manager REQUIRED
qgisMinimumVersion
e.g. 3.28 — the floor you actually test on REQUIRED
description
One short line; no line breaks REQUIRED
version
semver, e.g. 1.4.2 — must be unique per upload REQUIRED
author
Your name / org REQUIRED
email
Contact for the repo admins REQUIRED
supportsQt6
True once you're clean under Qt6 / QGIS 4
qgisMaximumVersion
Optional cap; omit to allow all 3.x (defaults to 3.99)
about
Longer paragraph shown on the plugin page
repository
Your GitHub URL — strongly recommended
tracker
Issues URL (e.g. GitHub Issues)
homepage
Docs / landing page
tags
Comma-separated, aids search
changelog
Multi-line notes for this version
experimental
True hides it from default installs — good for betas
icon
Path relative to the folder, e.g. icon.png

name=Floating Info Tool

Since you target 3 & 4 supportsQt6 is the single flag that tells the repo (and QGIS 4 users) your plugin is Qt6-safe. Set it only after the plugin actually loads clean under a Qt6 build — a stray from PyQt5 import or an exec_() call will bite you.

Package it

# from the parent directory of the plugin folder
zip -r floating_info_tool-1.4.2.zip floating_info_tool \
    -x "*.pyc" -x "*__pycache__*" -x "*.git*"
Nicer option pb_tool (pip install pb_tool) reads a pb_tool.cfg and gives you pb_tool zip — it compiles resources and excludes junk for you. Handy when you're shipping several plugins.
A

GitHub release

This is the canonical home of the source and the human-readable changelog. Do it first so the tag and the zip you attach are the exact bytes you send to the official repo.

  1. Commit the version bump
    Make sure metadata.txt version= matches the tag you're about to cut, then push. 
    git add metadata.txt CHANGELOG.md
git commit -m "Release 1.4.2"
git push
  2. Tag it — semver, v-prefixed
    A tag per release keeps GitHub Releases and the repo version in lockstep. 
    git tag -a v1.4.2 -m "Floating Info Tool 1.4.2"
git push origin v1.4.2
  3. Draft the release
    On GitHub: Releases → Draft a new release. Pick the tag v1.4.2, title it, and paste the changelog. Keep the notes user-facing — what changed, not how.
  4. Attach the zip as a release asset
    Drag floating_info_tool-1.4.2.zip into the assets box. This is the download users grab if they install outside Plugin Manager, and the artifact you upload in Track B.
  5. Publish
    Hit publish. The release URL is a stable link you can reference from your portfolio and from metadata.txt's repository field.
Automate both tracks qgis-plugin-ci turns a pushed tag into both a GitHub release and an upload to plugins.qgis.org from GitHub Actions. Given you already run Actions elsewhere, it's the natural upgrade once the manual flow is muscle memory — qgis-plugin-ci release TAG --github-token … --osgeo-username ….
B

plugins.qgis.org

The official repository that feeds Plugin Manager. First-time submissions pass an automated checker and then a human approval; after that, you're a trusted owner and updates go live immediately.

  1. Sign in
    Log in at plugins.qgis.org with your OSGeo / QGIS web account. No account yet? Register once — this same identity is what qgis-plugin-ci uses as --osgeo-username.
  2. Upload the zip
    Go to Share a plugin → Upload (/plugins/add/). Select the same zip you attached to the GitHub release. The checker validates structure, metadata.txt, and runs security scans on the code.
  3. Clear the checker
    If it complains, it's almost always metadata or a security lint (see Common failures below). Fix, re-zip, bump the version if needed, re-upload.
  4. Wait for approval — first version only
    A brand-new plugin lands in an approval queue and a repo admin reviews it before it's public. This is a one-time gate; be patient and make sure your email is monitored.
  5. Ship updates instantly
    Once you own an approved plugin, every later version you upload (with a new version=) publishes immediately — no re-review. Users see the update badge in Plugin Manager.
The version rule Every upload needs a unique, higher version=. You cannot re-upload 1.4.2 to fix a typo — the repo will reject a duplicate. Bump to 1.4.3, re-tag, re-release.
✓

Release checklist

The pre-flight you run every single time, in order. If all six are true, you're clear to ship.

  • Version bumped in metadata.txt and it's higher than the last upload.
  • Changelog written — both in metadata.txt and your GitHub release notes.
  • Loads clean in a fresh profile on your qgisMinimumVersion; if supportsQt6=True, verified on a Qt6 build too.
  • Zip has one top-level folder, no __pycache__, no .git, LICENSE present.
  • Tag pushed as vX.Y.Z and the GitHub release published with the zip attached.
  • Same zip uploaded to plugins.qgis.org and the checker is green.
!!

Common checker failures

The ones that actually stop a submission, and the fix for each.

QT6 / QGIS 4
Qt5-only APIs with supportsQt6 set

Bare PyQt5 imports, exec_(), or QRegExp break under Qt6. Route through qgis.PyQt, use exec(), and gate anything version-specific behind a compat shim. Only then flip the flag.

SECURITY LINT · B110
try / except / pass

The scanner flags silent excepts. Catch the specific exception and at minimum QgsMessageLog it, or add a # nosec only where a swallow is genuinely intended and justified.

STRUCTURE
Loose files at zip root

Zipping the contents instead of the folder. The archive must contain exactly one directory whose name matches the plugin. Re-run the zip from the parent dir.

MANIFEST
Missing / malformed metadata.txt

Absent required field, a line break inside description, or a duplicate version. Keep description single-line; move prose to about.

ENTRY POINT
No classFactory

__init__.py must define def classFactory(iface): returning your plugin instance. Without it QGIS can't instantiate the plugin.

LICENSING
Non-free or missing LICENSE

The official repo expects a GPL-compatible license. Include the LICENSE file in the zip and name the license in your docs.
